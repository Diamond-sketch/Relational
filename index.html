Online / Offline / Near-line Storages

  • Online storage
  • immediately available for a computer system
  • e.g., cache memory, main memory and magnetic disks that always connect to a computer system.
  • Offline storage
  • not immediately available and normally require some human intervention to become online
  • e.g., punched cards, floppy disks, Zip disks, flash memory devices, removable hard disk drives and magnetic tapes.
  • Near-line storage
  • not immediately available, but can be made online quickly (by a robotic arm) without human intervention
  • e.g., tape libraries and optical jukeboxes

Disk Format

  • Each disk has two surfaces where data is stored.
  • If a disk stores data on only one of its surface, it is called single-sided.
  • If a disk stores data on both surfaces, it is called double-sided.
  • Each surface is divided into multiple concentric circles of small width, called tracks.
  • Tracks are separated by inter-track gaps.
  • Each track is typically divided into multiple sectors, so-called disk blocks.
  • Sectors are separated by inter-sector gaps, so-called inter-block gaps.
  • Gaps are used to help identifying the beginning of sectors.

Slide 36

Disk Access Time

  • Seek time – time for the disk to position the head assembly at the proper cylinder.
  • Latency time (i.e., rotational delay) – time for the disk to rotate the beginning of the desired sector containing the block into position under the read/write head.
  • Transfer time – time for the disk to transfer blocks of data between disk and main memory.
  • i.e., transfer sectors of the block and any gaps between them.
  • Normally, much less than seek and latency time.
  • Bulk transfer: several consecutive sectors on the same track or cylinder are transferred.
  • Disk access time in order of milliseconds is quite high compared with CPU processing time.

Slide 38

Buffering

  • In CPUs, multiple processes are running concurrently either in an interleaved fashion (in case of single CPU) or in a parallel fashion (in case of multiple CPUs).
  • When a process requires data from disk, it needs to wait until the transfer of the data block is completed.
  • Buffering is the process of pre-loading data into memory
  • It performs independently of and in parallel to CPU processing when a process requests for I/O (e.g., data from the disk).
  • The pages in the memory reserved for temporarily storing such data blocks is referred to as a buffer.

Buffering

  • For concurrent processing, the usage of multiple buffers can increase the overall throughput and help prevents bottlenecks of I/O requests.
  • Double buffering
  • allows multiple buffers to be filled,
  • permits continuous reading or writing of data on consecutive disk blocks, and
  • eliminates the seek time and the rotation delay for all but the first block transfer.

File Allocation Techniques

  • File blocks are allocated to consecutive disk blocks.

  • uses appropriate insertion/deletion algorithms.

Non-Clustered and Clustered Indices

  • When the CREATE INDEX command has been invoked,
  • an additional file, called index file, is created
  • it contains the sorted value of the indexed column(s) (i.e., indexed field(s)) along with the pointer to the record they are originated from.
  • We can create many non-clustered indexes for each table.
  • When the CREATE CLUSTERED INDEX command has been invoked,
  • the data file is reorganized and becomes an ordered file in which records (i.e., tuples) are sorted on the indexed column(s) (i.e., indexed field(s)).
  • Thus, there can be only a single clustered index (i.e., a single ordered file) for every table.

Lock-based Concurrency Control Protocols

  • Main idea: While one transaction accesses a data item, no other transaction can modify that data item.
  • Implement the idea by using locks.
  • A transaction must hold a lock on a data item as long as it accesses
  • The transaction does not necessary to unlock data item immediately after its last access of that data item, since serializability may not be ensured.
  • Locking can take place at many levels e.g., database, table, page, row and field (attribute). (lock granularity)

Deadlock Prevention with Transaction Timestamps

  • A transaction timestamp TS(T) is a unique identifier assigned to each transaction T.
  • It is typically based on the order in which transactions are starts.
  • TS(T1) < TS(T2) means is T1 older than T2.
  • Two schemes to prevent deadlock: wait-die and wound-wait.
  • Wait-die:
  • If TS(Ti) < TS(Tj) then older transaction Ti is allowed to wait
  • Otherwise, abort Ti (younger die) and restart it later with the same timestamp.
  • Wound-wait
  • If TS(Ti) < TS(Tj) then abort younger transaction Tj (older wounds younger) and restart with the same timestamp
  • Otherwise, younger transaction Ti is allowed to wait.
  • Both schemes ends up aborting the younger of the two transactions that may involved in the deadlock.

Deadlock Prevention with Waiting Algorithms

  • No require transaction timestamps.
  • No waiting algorithm
  • If a transaction is unable to obtain a lock, it is immediately aborted and then restart after a certain time delay without checking whether a deadlock will actually occur or not.
  • No transaction ever waits; so no deadlock can occur.
  • But transactions may abort and restart needlessly.
  • Cautious waiting algorithm
  • Try to reduce the number of needless aborts/restarts.
  • Suppose Ti tries to lock an item X which is being locked by Tj with a conflicting lock
  • If Tj is not waiting for some other locked item, then Ti is allowed to wait
  • Otherwise, abort Ti.

Basic Timestamp Ordering

  • T issues a read(x)
  • If writeTS(x) > TS(T) then abort and rollback T and reject read(x) of T
  • If writeTS(x) < TS(T) then execute read(x) and set readTS(x) = max(TS(T), readTS(x))
  • T issues a write(x)
  • If readTS(x) > TS(T) or writeTS(x) > TS(T) then abort and rollback T and reject write(x) of T
  • Otherwise, execute write(x) and set writeTS(X) = TS(T)
  • Guarantee conflict serializable schedules based on chronological order of timestamps
  • But not guarantee recoverable schedules
