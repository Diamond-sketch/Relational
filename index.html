Exercise 1: Disk Capacity

↑
โจทย์ย่อ: 8 platters = 16 surfaces, 2^14 tracks/surface, 2^7 sectors/track, 2^12 bytes/sector. ถาม disk capacity, block size ถ้า 1 block = 4 sectors, และ blocks/track.
a) Disk capacity
สูตร: capacity = surfaces × tracks/surface × sectors/track × bytes/sector
  1. = 16 × 16,384 × 128 × 4,096
  2. = 2^4 × 2^14 × 2^7 × 2^12 = 2^37 bytes
  3. 2^37 bytes = 128 GiB
คำตอบ: 128 GiB หรือ 137,438,953,472 bytes
b) Block size
สูตร: block size = sectors/block × bytes/sector
  1. = 4 × 4,096
  2. = 16,384 bytes
คำตอบ: 16,384 bytes = 16 KiB
c) Blocks per track
สูตร: blocks/track = sectors/track ÷ sectors/block
  1. = 128 ÷ 4
  2. = 32
คำตอบ: 32 blocks/track

Exercise 2: Capacity & Data Access Time

↑
โจทย์ย่อ: B=512 bytes, G=128 bytes, 20 blocks/track, 400 tracks/surface, 15 double-sided disks = 30 surfaces, 2400 rpm, average seek=30 ms.
a) Track capacity
สูตร: track total = blocks/track × (B+G), useful = blocks/track × B
  1. total = 20 × (512+128) = 12,800 bytes
  2. useful = 20 × 512 = 10,240 bytes
คำตอบ: Track total = 12,800 bytes, useful = 10,240 bytes
b) Number of cylinders
สูตร: cylinders = tracks per surface
  1. There are 400 tracks per surface
  2. Tracks with same radius across surfaces form cylinders
คำตอบ: 400 cylinders
c) Cylinder capacity
สูตร: cylinder = track capacity × number of surfaces
  1. surfaces = 15 × 2 = 30
  2. total = 12,800 × 30 = 384,000 bytes
  3. useful = 10,240 × 30 = 307,200 bytes
คำตอบ: Cylinder total = 384,000 bytes, useful = 307,200 bytes
d) Disk pack capacity
สูตร: disk pack = cylinder capacity × cylinders
  1. total = 384,000 × 400 = 153,600,000 bytes
  2. useful = 307,200 × 400 = 122,880,000 bytes
คำตอบ: Total = 153.6 MB, useful = 122.88 MB
e-i) Transfer rate and block transfer time
สูตร: rotation time = 60,000/rpm; tr = track bytes/rotation time; btt = (B+G)/tr_total = B/tr_useful
  1. rotation time = 60,000/2400 = 25 ms
  2. total transfer rate = 12,800/25 = 512 bytes/ms
  3. useful transfer rate / bulk transfer rate = 10,240/25 = 409.6 bytes/ms
  4. btt = 640/512 = 1.25 ms
คำตอบ: tr(total)=512 B/ms, btr(useful)=409.6 B/ms, btt=1.25 ms
e-ii) Average rotational delay
สูตร: rd = one rotation time ÷ 2
  1. rd = 25/2
คำตอบ: 12.5 ms
e-iii) Bulk transfer rate
สูตร: bulk transfer rate = useful bytes per track ÷ rotation time
  1. = 10,240 ÷ 25
คำตอบ: 409.6 bytes/ms
f) Locate and transfer single block
สูตร: time = seek + rotational delay + block transfer time
  1. = 30 + 12.5 + 1.25
คำตอบ: 43.75 ms
g) 20 random vs 20 consecutive blocks
สูตร: random = 20×single block; consecutive double buffering = seek+rd+20×btt
  1. random = 20 × 43.75 = 875 ms
  2. consecutive = 30 + 12.5 + 20×1.25 = 67.5 ms
  3. Consecutive is much faster because seek and rotational delay happen only once
คำตอบ: Random ≈ 875 ms; consecutive ≈ 67.5 ms

Exercise 3: Number of Blocks Needed

↑
โจทย์ย่อ: 3,456 STUDENT records. Fields: 30, 9, 40, 10, 8, 1 bytes. Block size = 512 bytes. Unspanned.
a) Record size
สูตร: R = sum of all field sizes
  1. R = 30+9+40+10+8+1
คำตอบ: 98 bytes
b) Blocking factor and number of blocks
สูตร: bfr = floor(B/R); nBlocks = ceil(nRecords/bfr)
  1. bfr = floor(512/98) = 5 records/block
  2. nBlocks = ceil(3456/5) = ceil(691.2)
คำตอบ: bfr = 5, nBlocks = 692 blocks
c) Unused space
สูตร: unused/block = B − bfr×R; total unused = unused/block × nBlocks
  1. unused/block = 512 − 5×98 = 22 bytes
  2. total unused = 22 × 692 = 15,224 bytes
คำตอบ: 15,224 bytes ≈ 14.87 KiB

Exercise 4: Linear Search vs Binary Search

↑
โจทย์ย่อ: 20,000 STUDENT records. Fields total + 1 deletion marker. s=30 ms, rd=12.5 ms, btt=1 ms, btr=409.6 B/ms, B=512.
a) Record size R
สูตร: R = all fields + deletion marker
  1. R = 30+9+40+10+8+1+4+4+4+3+1
คำตอบ: 114 bytes
b) Blocking factor and file blocks
สูตร: bfr = floor(B/R); nBlocks = ceil(nRecords/bfr)
  1. bfr = floor(512/114) = 4 records/block
  2. nBlocks = ceil(20000/4)
คำตอบ: bfr = 4, nBlocks = 5,000 blocks
c-i) Average linear search, contiguous + double buffering
สูตร: avg blocks checked = nBlocks/2; time = s+rd+(avgBlocks×B/btr)
  1. avg blocks = 5000/2 = 2500
  2. transfer time = 2500×512/409.6 = 3125 ms
  3. total = 30+12.5+3125
คำตอบ: 3,167.5 ms
c-ii) Average linear search, not contiguous
สูตร: time = avgBlocks × (s+rd+btt)
  1. one random block = 30+12.5+1 = 43.5 ms
  2. total = 2500×43.5
คำตอบ: 108,750 ms
d) Binary search on ordered SSN
สูตร: accesses = ceil(log2 nBlocks); time = accesses × (s+rd+btt)
  1. accesses = ceil(log2 5000) = 13
  2. time = 13×43.5
คำตอบ: 565.5 ms

Exercise: Types of Index

↑
โจทย์ย่อ: Decide index type from data file organization and indexed field.
Rule 1) Primary index
สูตร: ordered file + ordering key field
  1. Index has one entry per data block
  2. Uses block pointer
  3. Sparse index
คำตอบ: Primary index = sparse
Rule 2) Clustering index
สูตร: ordered file + ordering non-key field
  1. One index entry per distinct value of clustering field
  2. Points to first block of each group
คำตอบ: Clustering index = sparse-like by distinct values
Rule 3) Secondary index
สูตร: non-ordering field or extra index on any file
  1. Usually dense
  2. One entry per record or per distinct search-key value with pointer list
คำตอบ: Secondary index = dense

Search tree / B-tree concept questions

↑
โจทย์ย่อ: Slides ask whether trees satisfy search-tree/B-tree rules.
Search tree rule
สูตร: Keys inside node sorted and child subtrees must follow key ranges
  1. Left subtree values < first key
  2. Middle subtree values between surrounding keys
  3. Right subtree values > last key
คำตอบ: A valid search tree must satisfy all range rules
B-tree rule
สูตร: Order n: max n tree pointers, max n−1 keys, internal nodes at least ceil(n/2) pointers, all leaves same level
  1. For order 3, each node has at most 3 pointers and 2 keys
  2. Non-root internal nodes need at least ceil(3/2)=2 pointers
  3. Leaves must be at same level
คำตอบ: Use these rules to check diagrams

  3. nLeaf ≤ 31.625

1) SELECT * FROM Staff WHERE salary > 1000
สูตร: selection σ
  1. Use σ_salary>1000(Staff)
คำตอบ: σ salary>1000 (Staff)
2) SELECT * FROM Staff WHERE salary>10000 AND NOT(position=Manager)
สูตร: selection with AND and NOT
  1. Combine predicates inside σ
คำตอบ: σ (salary>10000) ∧ ¬(position=Manager) (Staff)
3) SELECT staffNo,fName,lName,salary FROM Staff
สูตร: projection π
  1. Keep only selected columns
คำตอบ: π staffNo,fName,lName,salary (Staff)
4) SELECT columns FROM Staff WHERE salary>10000
สูตร: selection first, then projection
  1. Filter rows first
  2. Then keep required columns
คำตอบ: π staffNo,fName,lName,salary (σ salary>10000 (Staff))
5) WHERE position=Manager then project columns
สูตร: cannot project away position before selection
  1. If π removes position first, σ cannot check position
คำตอบ: π staffNo,fName,lName,salary (σ position=Manager (Staff))

  2. INTERSECT = ∩
  3. EXCEPT = −

7) Client × Viewing

  1. SQL: SELECT * FROM Client, Viewing

  1. Join condition Client.clientNo=Viewing.clientNo

  1. Keep all PropertyForRent rows

Query Optimization Strategy Cost

↑
โจทย์ย่อ: Query: Staff join Branch where Staff.branchNo=Branch.branchNo AND position=Manager AND city=London. Stats: 1000 Staff, 50 Managers, 50 Branches, 5 London.
Strategy I: Cross product then selection
สูตร: read Staff + read Branch + write cross product + read cross product
  1. = 1000 + 50 + (1000×50) + (1000×50)
คำตอบ: 101,050 disk accesses
Strategy II: Join then selection
สูตร: read Staff + read Branch + write join + read join
  1. = 1000 + 50 + 1000 + 1000
คำตอบ: 3,050 disk accesses
Strategy III: Select first, then join
สูตร: reduce each relation before join
  1. Read Staff and select Managers = 1000
  2. Write Managers = 50
  3. Read Branch and select London = 50
  4. Write London = 5
  5. Read reduced relations and join = 50
  6. Small final intermediate ≈ 5
  7. Total = 1000+50+50+5+50+5
คำตอบ: 1,160 disk accesses
Best strategy
สูตร: choose minimum cost
  1. I = 101,050
  2. II = 3,050
  3. III = 1,160
คำตอบ: Strategy III is best because selections are done early

Exercise: Draw query trees

↑
โจทย์ย่อ: Draw query trees for three relational algebra expressions.
A) σ over Staff × Branch
สูตร: tree root = σ, child = ×, leaves = Staff and Branch
  1. Root: σ position=Manager ∧ city=London ∧ Staff.branchNo=Branch.branchNo
  2. Child: ×
  3. Leaves: Staff, Branch
คำตอบ: Least efficient tree because cross product is large
B) σ over Staff ⋈ Branch
สูตร: tree root = σ, child = join
  1. Root: σ position=Manager ∧ city=London
  2. Child: ⋈ Staff.branchNo=Branch.branchNo
  3. Leaves: Staff, Branch
คำตอบ: Better than A because join replaces cross product
C) Join of two selections
สูตร: tree root = join, children = σ Staff and σ Branch
  1. Root: ⋈ Staff.branchNo=Branch.branchNo
  2. Left child: σ position=Manager(Staff)
  3. Right child: σ city=London(Branch)
คำตอบ: Best heuristic tree because selection is pushed down early

Example: Estimating SELECTION Cost

↑
โจทย์ย่อ: Given indexes and statistics: hash Staff.staffNo, clustering Staff.branchNo, B+-tree Staff.salary, n=3000, f=30, branch DV=500, position DV=10, salary range 10000–50000.
1) σ staffNo = SG5
สูตร: hash index equality on primary key
  1. Hash index without overflow gives 1 block access
  2. Primary key means cardinality 1
คำตอบ: Cost C=1 block, SC=1
2) σ position = Manager
สูตร: no index ⇒ linear search; SC = n/DV
  1. bStaff = n/f = 3000/30 = 100 blocks
  2. SC = 3000/10 = 300
คำตอบ: Cost C=100 blocks, SC=300
3) σ branchNo = B003
สูตร: clustering index cost = HTI + ceil(SC/f)
  1. SC = 3000/500 = 6
  2. C = 2 + ceil(6/30) = 3
คำตอบ: Cost C=3 blocks, SC=6
4) σ salary > 20000
สูตร: B+-tree range cost = HTI + bLI/2 + n/2; SC from range fraction
  1. C = 2 + 50/2 + 3000/2 = 1527
  2. SC = 3000×(50000−20000)/(50000−10000)
  3. = 2250
คำตอบ: Cost C=1527 blocks, SC=2250
5) σ position=Manager AND branchNo=B003
สูตร: use the more useful index first, then check remaining predicate
  1. Use clustering index on branchNo
  2. Cost = 3
  3. Intermediate nT = SC branchNo = 6
  4. DVT.position=(nT+DVposition)/3=(6+10)/3≈6
  5. SC = 6/6 = 1
คำตอบ: Cost C=3 blocks, SC≈1

Example: Estimating JOIN Cost

↑
โจทย์ย่อ: Given: buffer=100 blocks, Staff b=200 n=6000, Branch b=10 n=500, PropertyForRent b=2000 n=100000. Hash indexes on Staff.staffNo and Branch.branchNo.

  1. Block nested loop: C = bS + (bS×bP/(buffer−2)) = 200 + 200×2000/98 ≈ 4282

  4. Hash join: 3×(10+2000)=6030

Projection and Set Operation Cost

↑
โจทย์ย่อ: Formulas from slides for projection and set operations.
Projection cost
สูตร: remove non-required attributes + remove duplicates
  1. If sorting: C = bR + bR×log2(bR)
  2. If hashing: C = bR + nb
คำตอบ: Use sorting/hashing formula depending on algorithm
Projection cardinality
สูตร: S = πA(R)
  1. If A is key: nS=nR
  2. Otherwise: nS=SC_R.A
คำตอบ: Key keeps all tuples unique; non-key collapses duplicates
Set operation cost
สูตร: sort-merge UNION / INTERSECTION / DIFFERENCE
  1. C = bR+bS+bR×log2(bR)+bS×log2(bS)
คำตอบ: Use this for set operations when sort-merge is used
Set operation cardinality bounds
สูตร: use min/max bounds
  1. UNION: max(nR,nS) < nT < nR+nS
  2. INTERSECTION: 0 < nT < min(nR,nS)
  3. DIFFERENCE: 0 < nT < nR
คำตอบ: These are estimates/bounds, not exact unless data distribution is known

Physical Design and Tuning: “งานที่ต้องคิด” checklist

↑
โจทย์ย่อ: This lecture is mostly decision-making, not numeric calculation. Use these rules to answer index/tuning questions.
1) Choose attributes to index
สูตร: index candidates = attributes in selection/join conditions, keys, uniqueness constraints
  1. High-frequency queries matter most (80–20 rule)
  2. Strict time constraints make an attribute higher priority
  3. Candidate keys should have access paths
คำตอบ: Index WHERE/JOIN/key attributes first
2) Avoid too many indexes on frequently updated files
สูตร: update overhead increases with number of indexes
  1. INSERT/DELETE/UPDATE must maintain every index
  2. A table with 10 indexes has more update overhead than a table with 1–2 indexes
คำตอบ: For update-heavy tables, keep indexes minimal
3) Tree index vs hash index
สูตร: tree for equality + range; hash for equality only
  1. B+-tree supports =, >, <, BETWEEN, ORDER BY
  2. Hash is faster for exact equality but not range
คำตอบ: Use B+-tree for range, hash for equality
4) SQL index syntax
สูตร: CREATE [UNIQUE] INDEX name ON table(column [ASC/DESC], ...) [CLUSTER]
  1. UNIQUE when duplicate values are not allowed
  2. CLUSTER/clustered index sorts data records on that index attribute
คำตอบ: Example: CREATE INDEX idx_staff_branch ON Staff(branchNo);
5) Query tuning rules
สูตร: reduce data early and avoid expensive operations when possible
  1. Use SELECT columns instead of SELECT *
  2. Push selection before join
  3. Use indexed join attributes
  4. Avoid string comparison when another key works
  5. Avoid unnecessary DISTINCT, ORDER BY, and nested queries
คำตอบ: Goal = lower I/O cost and response time

Read-set / Write-set and ACID

↑
โจทย์ย่อ: Slides ask transaction concepts and how DBMS sees read/write operations.
Read-set / Write-set
สูตร: Read-set = all items read; Write-set = all items written
  1. Read-only transaction: write-set is empty
  2. Read-write transaction: at least one write operation
คำตอบ: Use sets like Read-set={X,Y}, Write-set={X}
ACID properties
สูตร: Atomicity, Consistency, Isolation, Durability
  1. Atomicity: all or nothing
  2. Consistency/serializability: correct database state
  3. Isolation: one transaction should not interfere with another
  4. Durability: committed data survives failure
คำตอบ: Remember A-C-I-D
Commit vs Rollback
สูตร: commit = successful end; rollback/abort = undo unsuccessful transaction
  1. If transaction completes, changes are committed
  2. If transaction fails, effects must be undone
คำตอบ: COMMIT keeps changes; ROLLBACK undoes changes

Concurrent Transaction Problems

↑
โจทย์ย่อ: Airline reservation example: T1 transfers N seats from X to Y, T2 reserves M seats for X, T3 sums all reserved seats.
Lost update problem
สูตร: two transactions write same item; one update overwrites the other
  1. T1 reads X and computes X−N
  2. T2 reads X and computes X+M
  3. Whichever writes last overwrites the first result
คำตอบ: Final X may miss either −N or +M
Temporary update / dirty read
สูตร: a transaction reads a value written by another uncommitted transaction
  1. T1 writes X
  2. T2 reads that X before T1 commits
  3. If T1 aborts, T2 used invalid data
คำตอบ: Dirty read causes incorrect result and rollback issues
Incorrect summary problem
สูตร: summary transaction reads mixed old/new values
  1. T3 sums X before transfer and Y after transfer, or vice versa
  2. The total becomes inconsistent although each read is individually valid
คำตอบ: Aggregation sees inconsistent snapshot

Recoverable, Cascadeless, Strict Schedules

↑
โจทย์ย่อ: Classify schedules by when transactions read/write/commit/abort.
Recoverable schedule
สูตร: If Tj reads data written by Ti, then Tj commits after Ti commits
  1. Commit order must respect read-from dependency
  2. If reader commits before writer, schedule is non-recoverable
คำตอบ: Recoverable prevents committed transaction from depending on aborted data
Cascading rollback
สูตร: reader reads uncommitted data and writer aborts
  1. T2 reads X from T1
  2. T1 aborts
  3. T2 must also abort
คำตอบ: This can force many transactions to rollback
Cascadeless schedule
สูตร: transactions read only committed data
  1. Delay read until writer commits
  2. Avoids cascading rollback
คำตอบ: Every read is from a committed transaction
Strict schedule
สูตร: no transaction may read or write X until the last writer of X commits/aborts
  1. Stronger than cascadeless
  2. Prevents dirty reads and dirty writes
คำตอบ: Strict schedule is safest for recovery

Serializability and Conflict Serializable Examples

↑
โจทย์ย่อ: Decide whether non-serial schedules are equivalent to a serial schedule.
Serial vs non-serial
สูตร: serial = all operations of each transaction run consecutively; non-serial = interleaved
  1. Serial is always correct but slower
  2. Non-serial can be faster but must be checked
คำตอบ: Need serializability to know if non-serial is correct
Conflict rule
สูตร: operations conflict if different transactions + same item + at least one write
  1. r1(X) and r2(X) do not conflict
  2. r1(X) and w2(X) conflict
  3. w1(X) and w2(X) conflict
คำตอบ: Use conflicts to build precedence graph
Testing serializability
สูตร: build graph: Ti → Tj if an operation of Ti conflicts before operation of Tj
  1. If graph has a cycle: not conflict serializable
  2. If no cycle: conflict serializable
  3. Topological order gives equivalent serial order
คำตอบ: No cycle = serializable
Schedule C / D idea
สูตร: check conflicts on X between T1 and T2
  1. If graph has edges both ways, cycle occurs
  2. If graph has one direction only, equivalent to one serial order
คำตอบ: Slide result: D is serializable and equivalent to A
Schedule E example
สูตร: precedence graph has T3 → T1 → T2 pattern
  1. Slide graph has no cycle
  2. Topological order is T3 then T1 then T2
คำตอบ: Schedule E is serializable; equivalent serial order = T3 → T1 → T2
