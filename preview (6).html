Disk Format

  • Each disk has two surfaces where data is stored.
  • If a disk stores data on only one of its surface, it is called single-sided.
  • If a disk stores data on both surfaces, it is called double-sided.

Solution / answer
Disk Format answer

A disk has two surfaces. Single-sided uses one surface; double-sided uses both. Surfaces are divided into tracks. Tracks are separated by inter-track gaps. Tracks are divided into sectors / disk blocks. Sectors are separated by inter-sector gaps / inter-block gaps. Gaps help identify the beginning of sectors.

Slide 36

Disk Access Time

  • Seek time – time for the disk to position the head assembly at the proper cylinder.
  • Latency time (i.e., rotational delay) – time for the disk to rotate the beginning of the desired sector containing the block into position under the read/write head.
  • Transfer time – time for the disk to transfer blocks of data between disk and main memory.
  • i.e., transfer sectors of the block and any gaps between them.
  • Normally, much less than seek and latency time.
Solution / answer
Disk Access Time answer

Disk access time = seek time + rotational latency/delay + transfer time. Seek positions the head at the proper cylinder. Latency waits for the sector. Transfer time moves blocks between disk and main memory.

Slide 38

Buffering

  • In CPUs, multiple processes are running concurrently either in an interleaved fashion (in case of single CPU) or in a parallel fashion (in case of multiple CPUs).
  • When a process requires data from disk, it needs to wait until the transfer of the data block is completed.
  • Buffering is the process of pre-loading data into memory
  • It performs independently of and in parallel to CPU processing when a process requests for I/O (e.g., data from the disk).
  • The pages in the memory reserved for temporarily storing such data blocks is referred to as a buffer.
Slide 39

Buffering

  • For concurrent processing, the usage of multiple buffers can increase the overall throughput and help prevents bottlenecks of I/O requests.
  • Double buffering
  • allows multiple buffers to be filled,
  • permits continuous reading or writing of data on consecutive disk blocks, and
  • eliminates the seek time and the rotation delay for all but the first block transfer.

Motivation

  • All index levels of the multilevel index that we have seen are physically ordered file.
  • Insertions and deletions of a record in the physically ordered file is complex.
  • To retain the benefits of using multilevel indexing while reducing index insertion and deletion problems, designers adopted a dynamic multilevel index that
  • leaves some space in each of its blocks for inserting new entries and
  • uses appropriate insertion/deletion algorithms.

Physical Operation

  • Physical operation refers to a specific algorithm that implements a logical database operation (e.g, selection or join)
  • Replacing the logical operations in a relational algebra tree with physical operations produces and execution strategy (a.k.a., query execution plan, query evaluation plan or access plan).
  • Different DBMS can have different physical operations and different execution strategies.
Slide 105

Basic Physical Operations

  • Table-scan(R) refers to accessing all blocks of R in an arbitrary order.
  • Index-scan (R, A) refers to accessing blocks of the index file of an attribute A of relation R to search for tuples (that a satisfy a condition).
  • Index-scan (R, p) refers to accessing blocks of the index file of an attribute of relation R specified in predicate p to search for tuples (that a satisfy the predicate p).
  • Sort-scan (R, L) refers to accessing blocks of (unsorted or sorted, indexed or non-indexed) relation R and producing blocks of the sorted relation on a specific attribute in list L.

Sorting

  • Sorting records in large files stored on disk is referred to as external sorting.
  • Sorting records stored in main-memory is referred to as internal sorting.
  • Normally, external sorting uses sort-merge strategy that starts by sorting small sub-files, so-called runs, of the main file and then merges the sorted runs, creating larger sorted sub-files that are merged in turn.

Why do transactions need to be managed?

  • Parallel processing (on multiple CPUs)
  • When transactions are executing concurrently (in an interleaved fashion), the order of execution of operations from all the various transactions is known as a schedule (a.k.a. history).

Why do transactions need to be managed?

  • Lost update problem
  • Temporary update problem (a.k.a. read uncommitted data or dirty read problem)
  • Incorrect summary problem (i.e. inconsistent retrieval problem)
  • To guarantee correct results, DBMS needs to control execution of operations from those concurrent transactions which is known as a concurrency control.
